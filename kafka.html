Что такое Kafka?

Apache Kafka — это распределённая платформа для потоковой передачи данных. Проще говоря — это очень быстрая и надёжная очередь сообщений, в которую одни сервисы пишут, а другие читают.

💡 Аналогия

Представь Kafka как ленту новостей в соцсети. Один человек публикует пост (Producer), а сотни подписчиков читают его в удобное для них время (Consumer). Kafka хранит все посты в хронологическом порядке.

⚠️ Когда Kafka НЕ нужна

Kafka — это не серебряная пуля. Если у тебя маленький проект и нагрузка в несколько запросов в секунду — достаточно простой очереди (RabbitMQ, Redis Streams или даже просто БД).

Урок 02 · Основы
Ключевые концепции Kafka
Разбираем терминологию: Topic, Producer, Consumer, Broker, Partition, Offset. Без этого дальше двигаться сложно.

Основные понятия

📌
Topic
Категория/канал, в который пишутся сообщения. Как таблица в БД, но для потоков.
📤
Producer
Приложение, которое отправляет сообщения в topic.
📥
Consumer
Приложение, которое читает сообщения из topic.
🖥️
Broker
Сервер Kafka. Хранит данные и обслуживает запросы.
📦
Partition
Раздел внутри topic. Позволяет параллельно обрабатывать данные.
🔢
Offset
Порядковый номер сообщения внутри partition (0, 1, 2...). Как ID строки.
👥
Consumer Group
Группа consumer-ов, совместно читающих один topic, распределяя нагрузку.
📋
Message
Единица данных в Kafka. Содержит key, value, timestamp, headers.

Topic — главная абстракция

Все данные в Kafka организованы в topics. Это как папки: в одну папку складывают логи, в другую — заказы, в третью — клики пользователей.

📌 Примеры названий topic-ов

user-registrations  order-events  payment-completed  page-views  sensor-data

Сообщение (Message)

Каждое сообщение в Kafka — это просто набор байт. Обычно используют форматы JSON, Avro или Protobuf.

{
  "key": "user-123",       // ключ для распределения по партициям
  "value": {            // полезная нагрузка
    "event": "order_created",
    "orderId": "ord-456",
    "amount": 1500.00
  },
  "timestamp": 1703001600000,
  "headers": {          // метаданные
    "source": "order-service"
  }
}

Partition и Offset — как хранятся данные

Каждый topic разбивается на несколько partition (разделов). Внутри каждого partition сообщения пронумерованы — это и есть offset.

Partition 0
offset: 0 | "Заказ 1"
offset: 1 | "Заказ 4"
offset: 2 | "Заказ 7"
Partition 1
offset: 0 | "Заказ 2"
offset: 1 | "Заказ 5"
offset: 2 | "Заказ 8"
Partition 2
offset: 0 | "Заказ 3"
offset: 1 | "Заказ 6"
⚠️ Порядок сообщений

Kafka гарантирует порядок сообщений только внутри одной partition. Между разными partition порядок не гарантирован. Поэтому если важен порядок — используй один ключ (сообщения с одинаковым ключом попадают в одну partition).

Broker — сервер Kafka

Broker — это один узел (сервер) в кластере Kafka. Он принимает сообщения от Producer-ов, хранит их и отдаёт Consumer-ам. В продакшне обычно 3+ broker-а для отказоустойчивости.

⚡ Проверка знаний
Ты хочешь, чтобы все события одного пользователя обрабатывались строго по порядку. Что нужно сделать?
Создать больше topic-ов
Увеличить количество broker-ов
Использовать user_id как ключ сообщения — тогда все его события попадут в одну partition
Уменьшить количество partition до 1 для всего topic
Урок 03 · Архитектура
Как устроена Kafka изнутри
Кластер, репликация, ZooKeeper vs KRaft — разбираем архитектуру Kafka и понимаем почему она такая надёжная.

Кластер Kafka

В продакшне Kafka работает как кластер — несколько broker-ов объединены вместе. Это даёт отказоустойчивость и масштабируемость.

Кластер из 3 broker-ов

Broker 1 (Leader)
⭐ Partition 0
Partition 1 (replica)
Partition 2 (replica)
Broker 2
⭐ Partition 1
Partition 0 (replica)
Partition 2 (replica)
Broker 3
⭐ Partition 2
Partition 0 (replica)
Partition 1 (replica)

Репликация данных

Каждая partition хранится на нескольких broker-ах. Один из них становится Leader — он принимает все записи и чтения. Остальные — Followers, они копируют данные с Leader-а.

Если Leader упадёт — один из Follower-ов автоматически станет новым Leader-ом. Данные не потеряются!

✅ Replication Factor

Replication Factor — сколько копий хранится. Обычно ставят 3:
replication.factor=3
Это значит данные выживут, даже если 2 из 3 серверов упадут.

ZooKeeper vs KRaft

Раньше Kafka использовала Apache ZooKeeper для хранения метаданных кластера (кто Leader, сколько broker-ов и т.д.). Это создавало дополнительную сложность.

ZooKeeper (старый способ)KRaft (новый, с Kafka 3.0)
СложностьНадо поднять отдельный ZooKeeper кластерВсё встроено в Kafka
МасштабДо ~200 000 partitionМиллионы partition
Скорость стартаМедленнееБыстрее
РекомендацияУстаревший, скоро удалят✅ Используй в новых проектах

Жизненный цикл сообщения

1
Producer отправляет сообщение
Producer выбирает topic и (опционально) ключ. Kafka определяет в какую partition попадёт сообщение на основе ключа.
2
Broker-Leader записывает сообщение
Leader partition записывает сообщение в лог (append-only файл на диске). Присваивает offset.
3
Репликация на Followers
Follower broker-ы копируют сообщение. После подтверждения от нужного количества реплик — сообщение считается "закоммиченным".
4
Consumer читает сообщение
Consumer подписывается на topic, Kafka отдаёт сообщения начиная с нужного offset. Consumer сохраняет (commit) свой offset — чтобы знать, что уже прочитал.
⚡ Проверка знаний
Один из трёх broker-ов в кластере упал. Что произойдёт с данными?
Данные потеряются — надо было делать бэкап
Данные сохранятся — они реплицированы на других broker-ах, один из Followers станет Leader
Kafka полностью остановится до восстановления сервера
Producer-ы перестанут работать, но Consumer-ы продолжат
Урок 04 · Практика
Установка и первые шаги
Запускаем Kafka локально через Docker. Создаём topic и отправляем первые сообщения через командную строку.

Вариант 1: Docker (самый простой)

Самый быстрый способ — один контейнер с KRaft режимом (без ZooKeeper):

# Скачать и запустить Kafka в KRaft режиме
docker run -d \
  --name kafka \
  -p 9092:9092 \
  -e KAFKA_NODE_ID=1 \
  -e KAFKA_LISTENER_SECURITY_PROTOCOL_MAP='CONTROLLER:PLAINTEXT,PLAINTEXT:PLAINTEXT' \
  -e KAFKA_ADVERTISED_LISTENERS='PLAINTEXT://localhost:9092' \
  -e KAFKA_PROCESS_ROLES='broker,controller' \
  -e KAFKA_CONTROLLER_QUORUM_VOTERS='1@localhost:9093' \
  apache/kafka:latest

Вариант 2: Docker Compose (рекомендуется)

Более удобный способ — создай файл docker-compose.yml:

version: '3'
services:
  kafka:
    image: apache/kafka:latest
    ports:
      - "9092:9092"
    environment:
      KAFKA_NODE_ID: 1
      KAFKA_PROCESS_ROLES: broker,controller
      KAFKA_LISTENERS: PLAINTEXT://0.0.0.0:9092,CONTROLLER://0.0.0.0:9093
      KAFKA_ADVERTISED_LISTENERS: PLAINTEXT://localhost:9092
      KAFKA_CONTROLLER_QUORUM_VOTERS: "1@localhost:9093"
      KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: PLAINTEXT:PLAINTEXT,CONTROLLER:PLAINTEXT
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1

  kafka-ui:           # Удобный веб-интерфейс
    image: provectuslabs/kafka-ui:latest
    ports:
      - "8080:8080"
    environment:
      KAFKA_CLUSTERS_0_NAME: local
      KAFKA_CLUSTERS_0_BOOTSTRAPSERVERS: kafka:9092
    depends_on:
      - kafka
# Запустить
docker-compose up -d

# Проверить что работает
docker ps
🎨 Kafka UI

После запуска открой http://localhost:8080 — там будет визуальный интерфейс для управления Kafka. Очень удобно для изучения!

Урок 05 · Практика
Producer — отправка сообщений
Пишем Producer на Python и Java. Разбираем конфигурацию, acks, compression и батчинг.

Producer на Python

Установим библиотеку:

pip install kafka-python-ng

Простой producer:

from kafka import KafkaProducer
import json

# Создаём producer
producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    # Сериализация: Python dict → JSON → bytes
    value_serializer=lambda v: json.dumps(v).encode('utf-8'),
    key_serializer=lambda k: k.encode('utf-8') if k else None
)

# Отправляем сообщение
producer.send(
    topic='orders',
    key='user-123',       # ключ (необязательно)
    value={
        'order_id': 'ord-789',
        'user_id': 'user-123',
        'amount': 2500.0,
        'status': 'created'
    }
)

# Ждём подтверждения отправки всех сообщений
producer.flush()
producer.close()
print("Сообщение отправлено!")

Callback — обработка результата

def on_send_success(record_metadata):
    print(f"✅ Отправлено в topic: {record_metadata.topic}")
    print(f"   Partition: {record_metadata.partition}")
    print(f"   Offset: {record_metadata.offset}")

def on_send_error(exc):
    print(f"❌ Ошибка: {exc}")

producer.send('orders', value={'test': 1}) \
        .add_callback(on_send_success) \
        .add_errback(on_send_error)

Важные настройки Producer

ПараметрЗначениеЧто делает
acks0 / 1 / allСколько broker-ов должны подтвердить запись
retries0..MAX_INTСколько раз повторять при ошибке
batch_size16384 (байт)Размер батча для отправки
linger_ms0..5Ждать N мс перед отправкой батча
compression_typenone/gzip/snappy/lz4Сжатие сообщений
💡 acks — самый важный параметр
  • acks=0 — "Огонь и забудь". Максимальная скорость, нет гарантий. Можно потерять данные.
  • acks=1 — Ждём подтверждения только от Leader. Может потерять данные при падении Leader.
  • acks=all — Ждём от всех реплик. Максимальная надёжность, чуть медленнее.

Producer с оптимальными настройками

producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8'),
    
    # Надёжность
    acks='all',          # ждём подтверждения от всех реплик
    retries=3,           # 3 попытки при ошибке
    
    # Производительность
    batch_size=16384,    # батч до 16KB
    linger_ms=5,         # ждём 5мс для накопления батча
    compression_type='snappy'  # сжимаем данные
)
⚡ Проверка знаний
Ты разрабатываешь систему мониторинга, где важна скорость и потеря нескольких метрик не критична. Какой acks выбрать?
acks=0 или acks=1 — скорость важнее надёжности
acks=all — всегда нужна максимальная надёжность
Acks не влияет на скорость
Урок 06 · Практика
Consumer — чтение сообщений
Пишем Consumer на Python. Разбираем commit offset, auto.offset.reset и graceful shutdown.

Простой Consumer на Python

from kafka import KafkaConsumer
import json

consumer = KafkaConsumer(
    'orders',               # подписываемся на topic
    bootstrap_servers=['localhost:9092'],
    group_id='order-processor',  # имя группы
    
    # Десериализация: bytes → JSON → Python dict
    value_deserializer=lambda m: json.loads(m.decode('utf-8')),
    key_deserializer=lambda k: k.decode('utf-8') if k else None,
    
    # С какого места читать если group_id не знает этот topic:
    # 'earliest' = с начала, 'latest' = только новые
    auto_offset_reset='earliest',
    
    enable_auto_commit=True   # автоматически сохранять offset
)

print("Ожидаем сообщения...")
for message in consumer:
    print(f"\n📨 Новое сообщение:")
    print(f"   Topic:     {message.topic}")
    print(f"   Partition: {message.partition}")
    print(f"   Offset:    {message.offset}")
    print(f"   Key:       {message.key}")
    print(f"   Value:     {message.value}")

Ручной commit offset — для надёжности

По умолчанию Consumer автоматически сохраняет offset. Но что если сообщение прочиталось, но обработка упала? Данные будут помечены как обработанные, хотя это не так.

Решение — ручной commit после успешной обработки:

consumer = KafkaConsumer(
    'orders',
    bootstrap_servers=['localhost:9092'],
    group_id='order-processor-reliable',
    value_deserializer=lambda m: json.loads(m.decode('utf-8')),
    enable_auto_commit=False  # ❌ отключаем авто-commit
)

for message in consumer:
    try:
        # Обрабатываем сообщение
        order = message.value
        process_order(order)
        
        # ✅ Только после успешной обработки — commit
        consumer.commit()
        
    except Exception as e:
        print(f"❌ Ошибка обработки: {e}")
        # НЕ делаем commit — сообщение будет перечитано

Graceful Shutdown

Правильный способ остановить Consumer — обработать сигнал SIGINT (Ctrl+C):

import signal
import sys

running = True

def signal_handler(sig, frame):
    global running
    running = False
    print("\n🛑 Останавливаем consumer...")

signal.signal(signal.SIGINT, signal_handler)

while running:
    messages = consumer.poll(timeout_ms=1000)
    for tp, msgs in messages.items():
        for msg in msgs:
            process(msg)
            consumer.commit()

consumer.close()
print("✅ Consumer закрыт корректно")
⚠️ auto_offset_reset

earliest — читать с самого начала topic (если это новая группа)

latest — читать только новые сообщения (пришедшие после запуска consumer-а)

⚡ Проверка знаний
Consumer прочитал сообщение, начал его обрабатывать, но процесс упал до коммита offset. Что произойдёт при перезапуске?
Сообщение потеряется навсегда
Сообщение будет перечитано заново — offset не был сохранён

Partitions & Offsets
Понимаем как работает параллельность в Kafka, как выбирается partition и что такое retention policy.

Как Producer выбирает partition?

Когда Producer отправляет сообщение — Kafka решает, в какую partition его положить:

1
Указан ключ (Key)
partition = hash(key) % num_partitions
Одинаковый ключ → всегда одна и та же partition. Гарантирует порядок для одного пользователя/сущности.
2
Ключ не указан
Sticky Partitioner: накапливает батч в одну partition, потом переключается на следующую. Эффективно для больших объёмов.
3
Кастомный Partitioner
Можно написать свою логику: например, VIP-пользователей в отдельную partition для приоритетной обработки.

Retention Policy — как долго хранятся данные

В отличие от традиционных очередей, Kafka хранит сообщения долго. Можно настроить:

ПараметрПримерОписание
retention.ms604800000 (7 дней)Хранить сообщения N миллисекунд
retention.bytes1073741824 (1 GB)Максимальный размер partition
cleanup.policydelete / compactЧто делать со старыми данными
💡 Log Compaction

При cleanup.policy=compact Kafka хранит только последнее сообщение для каждого ключа. Идеально для хранения текущего состояния (например, последний статус заказа). Старые значения удаляются, последнее — всегда доступно.

Увеличение количества partition

# Посмотреть текущее количество
kafka-topics.sh \
  --bootstrap-server localhost:9092 \
  --describe \
  --topic orders

# Увеличить до 6 partitions
kafka-topics.sh \
  --bootstrap-server localhost:9092 \
  --alter \
  --topic orders \
  --partitions 6
⚠️ Нельзя уменьшить количество partition

Kafka позволяет только увеличивать количество partition. Уменьшить нельзя — это нарушит порядок сообщений. Если нужно уменьшить — придётся пересоздать topic и перенести данные.

Сколько partition нужно?

Общее правило:

  • Больше partition → выше параллельность, но больше нагрузка на broker
  • Хорошее число: от 3 до 12 для большинства случаев
  • Правило: число partition ≥ число consumer-ов в группе (иначе часть consumer-ов будет простаивать)
⚡ Проверка знаний
У тебя topic с 6 partition и нужно хранить только последнее состояние каждого объекта (например, текущий статус устройства). Что настроить?
retention.ms=-1 (хранить вечно)
cleanup.policy=compact — хранит только последнее сообщение по каждому ключу
retention.bytes=0 (не хранить)
Уменьшить количество partition до 1

Consumer Groups
Как несколько Consumer-ов делят работу между собой, что такое Rebalance и как мониторить отставание (lag).

Что такое Consumer Group?

Consumer Group — это несколько Consumer-ов, которые совместно читают один topic. Kafka автоматически распределяет partition между ними.

Topic с 6 partition — 3 consumer-а в группе

Partition
P0, P1
→
Consumer 1
order-svc-1
Partition
P2, P3
→
Consumer 2
order-svc-2
Partition
P4, P5
→
Consumer 3
order-svc-3
💡 Правила распределения
  • Каждая partition читается только одним Consumer-ом в группе
  • Один Consumer может читать несколько partition
  • Если Consumer-ов больше чем partition — лишние будут простаивать
  • Разные группы читают topic независимо и не мешают друг другу

Несколько групп = независимые читатели

Одни и те же данные можно читать разными командами независимо:

Topic
orders
→
Group: email-service
→
Group: analytics
→
Group: warehouse

Rebalance — перераспределение partition

Когда Consumer входит или выходит из группы (упал, добавился новый) — происходит Rebalance. Kafka перераспределяет partition между оставшимися Consumer-ами.

⚠️ Rebalance — это больно

Во время Rebalance все Consumer-ы в группе останавливаются. Это называется "stop the world". Чтобы минимизировать Rebalance: настраивай session.timeout.ms и max.poll.interval.ms разумно, используй Static Membership (Kafka 2.4+).

Consumer Lag — отставание

Consumer Lag — это разница между последним offset-ом в partition и текущим offset-ом Consumer-а. Показывает, насколько Consumer отстаёт от Producer-а.

# Посмотреть lag для всех групп
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe \
  --all-groups

# Для конкретной группы
kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --describe \
  --group order-processor

# Вывод будет примерно такой:
# GROUP           TOPIC   PARTITION  CURRENT-OFFSET  LOG-END-OFFSET  LAG
# order-processor orders  0          1050            1055            5
# order-processor orders  1          2000            2100            100 ← проблема!
⚡ Проверка знаний
У topic 4 partition. Ты запустил 6 consumer-ов в одной группе. Сколько будут работать?
Все 6 — Kafka раздаст каждому по partition
Только 4 — по одному на partition, 2 будут простаивать
Kafka вернёт ошибку — нельзя больше consumer-ов чем partition
Только 1 — Kafka выбирает одного лидера
Урок 09 · Углублённо
Надёжность данных
Репликация, семантика доставки (at-least-once, exactly-once), идемпотентность и транзакции.

Три семантики доставки

В распределённых системах существует три подхода к гарантии доставки сообщений:

СемантикаОписаниеКогда использовать
At-most-once Сообщение доставится 0 или 1 раз. Может потеряться. Метрики, логи, где потеря не критична
At-least-once Сообщение доставится минимум 1 раз. Может дублироваться. Большинство систем. По умолчанию в Kafka.
Exactly-once Сообщение доставится ровно 1 раз. Без потерь и дублей. Финансовые транзакции, критичные данные

Идемпотентный Producer

Чтобы избежать дублей при повторных отправках (например, при retry после ошибки сети) — включи идемпотентность:

producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    
    # Включаем идемпотентность
    enable_idempotence=True,  # автоматически установит:
                                  # acks='all'
                                  # retries=MAX
                                  # max_in_flight_requests_per_connection=5
)
💡 Как работает идемпотентность

Kafka присваивает каждому Producer уникальный ID (PID) и нумерует сообщения (Sequence Number). Если broker получает дубль — он его отбрасывает. Producer получает "успех", данные не дублируются.

Exactly-Once с транзакциями

Для полного exactly-once — используй транзакции. Это позволяет атомарно читать, обрабатывать и записывать данные:

producer = KafkaProducer(
    bootstrap_servers=['localhost:9092'],
    transactional_id='my-transactional-producer'
)

producer.init_transactions()

try:
    producer.begin_transaction()
    
    # Атомарная операция: всё или ничего
    producer.send('output-topic', value=processed_data)
    producer.send_offsets_to_transaction(
        offsets, consumer.consumer_group_id
    )
    
    producer.commit_transaction()
    
except Exception:
    producer.abort_transaction()  # откат

ISR — In-Sync Replicas

ISR — список реплик, которые синхронизированы с Leader-ом. Именно ISR реплики могут стать новым Leader-ом при сбое.

⚠️ min.insync.replicas

Настройка min.insync.replicas=2 гарантирует, что запись считается успешной только если минимум 2 реплики её подтвердили. В связке с acks=all это даёт максимальную надёжность.

⚡ Проверка знаний
Ты обрабатываешь финансовые транзакции. Одна и та же транзакция не должна быть обработана дважды. Что использовать?
At-most-once семантику — лучше потерять, чем задвоить
At-least-once с проверкой дублей в БД
Exactly-once через транзакции и идемпотентный Producer
Kafka не поддерживает финансовые транзакции
Урок 10 · Продвинутый
Kafka Streams
Обработка потоков данных в реальном времени прямо внутри Kafka — без Spark и Flink.

Что такое Kafka Streams?

Kafka Streams — это Java-библиотека для обработки потоков данных. Читает из topic, трансформирует данные и записывает результат в другой topic.

💡 Зачем нужен Kafka Streams?

Вместо: "Прочитай сообщения → обработай в своём коде → запиши результат" ты описываешь pipeline трансформаций декларативно. Kafka Streams масштабируется, отказоустойчив и не требует отдельного кластера (как Spark).

Основные операции

🔍
filter()
Оставить только сообщения, удовлетворяющие условию
🔄
map()
Трансформировать каждое сообщение
🔗
join()
Объединить два потока данных
📊
aggregate()
Агрегировать данные по ключу
⏰
windowing()
Группировать по временным окнам (последние 5 минут)
📋
KTable
Поток как таблица текущего состояния (changelog)

Пример на Java

// Считаем количество заказов по статусам в реальном времени
StreamsBuilder builder = new StreamsBuilder();

// Читаем поток заказов
KStream<String, Order> orders = builder
    .stream("orders");

// Считаем заказы по статусу
KTable<String, Long> orderCounts = orders
    .filter((key, order) -> order.amount > 0)   // фильтр
    .selectKey((key, order) -> order.status)       // группируем по статусу
    .groupByKey()
    .count();                                      // считаем

// Записываем результат
orderCounts
    .toStream()
    .to("order-counts");

// Запускаем
KafkaStreams streams = new KafkaStreams(builder.build(), config);
streams.start();

Windowing — временные окна

Одна из мощных функций — агрегация за временной период:

// Считаем сумму заказов за каждые 5 минут
orders
    .groupBy((key, order) -> order.userId)
    .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(5)))
    .aggregate(
        () -> 0.0,                              // начальное значение
        (userId, order, total) -> total + order.amount
    )
    .toStream()
    .to("user-revenue-per-5min");
✅ Ksql — SQL для Kafka

Не хочешь писать Java? Используй ksqlDB — можно обрабатывать потоки с помощью SQL:

SELECT status, COUNT(*) FROM orders GROUP BY status EMIT CHANGES;

Урок 11 · Продвинутый
Kafka Connect
Интеграция Kafka с базами данных, S3, Elasticsearch и другими системами без написания кода.

Что такое Kafka Connect?

Kafka Connect — фреймворк для интеграции Kafka с внешними системами. Позволяет без написания Producer/Consumer кода переносить данные между Kafka и:

  • Базами данных (PostgreSQL, MySQL, MongoDB)
  • Файловыми хранилищами (S3, GCS, HDFS)
  • Поисковыми движками (Elasticsearch, Solr)
  • Стриминговыми сервисами и CRM системами
📥
Source Connector
Читает из внешней системы → пишет в Kafka topic
📤
Sink Connector
Читает из Kafka topic → пишет во внешнюю систему

Пример: CDC с PostgreSQL

CDC (Change Data Capture) — отслеживание изменений в БД. Debezium Connector читает WAL-лог PostgreSQL и пишет каждое изменение в Kafka:

{
  "name": "postgres-source",
  "config": {
    "connector.class": "io.debezium.connector.postgresql.PostgresConnector",
    "database.hostname": "localhost",
    "database.port": "5432",
    "database.user": "kafka_user",
    "database.password": "secret",
    "database.dbname": "shop",
    "table.include.list": "public.orders,public.users",
    "topic.prefix": "db"
    // Создаст topics: db.public.orders, db.public.users
  }
}

Пример: Sink в Elasticsearch

{
  "name": "elasticsearch-sink",
  "config": {
    "connector.class": "io.confluent.connect.elasticsearch.ElasticsearchSinkConnector",
    "topics": "orders",
    "connection.url": "http://elasticsearch:9200",
    "type.name": "_doc",
    "key.ignore": "true"
  }
}

Типичные ошибки новичков

❌
Слишком мало partition на старте
Потом придётся добавлять и это нарушит порядок сообщений. Планируй на вырост: 6-12 partition на topic — хорошее начало.
❌
enable_auto_commit=True в критичных системах
Если сервис упадёт после commit но до реальной обработки — данные потеряются. Используй ручной commit.
❌
Один broker в продакшне
Это single point of failure. Минимум 3 broker-а для отказоустойчивости.
❌
Огромные сообщения
Kafka оптимизирована под маленькие сообщения (до 1 MB). Большие файлы — сохраняй в S3/MinIO, в Kafka пиши только ссылку.
❌
Не настроены алерты на Consumer Lag
Если lag растёт — Consumer не справляется. Нужно добавить consumer-ов или оптимизировать обработку.

Продакшн чеклист

✅ Минимальный чеклист для продакшна

  • ☐ replication.factor=3, min.insync.replicas=2
